Unit 4: Optimizing Content for Fast User Experience

Overview of Courses and What I Learned

  • Udacity Course: Web Performance Optimization

    This course was taught by Ilya Grigorik. He really impressed me. He now works for Google as a Developer Advocate, specifically advocating best practices for fast performance on HTTP networks, having built up a lot of experience with the network technology behind the internet. He even wrote an O'Reilly book on High-Performance Browser Networking. His blog archives are always great for web developers to read when they have time to go deep on technical issues, and his Twitter feed is always tells you what he considers to be the latest good information on such issues.

    This course showed us an excellent learning portal for web developers built by Google. It has many hours of content to learn from, surveying very many best practices for developers.

    Thanks to this course, I am now also comfortable with the vendor documentation for "Dev Tools" utilities both in Chrome and in Firefox.

    Finally, this course also introduced us to the PageSpeed Insights tool, which runs a battery of automated test on any page you are developing to help you make sure you aren't inadvertently and needlessly creating networking bottlenecks for delivering pages to users. This is especially of concern for mobile-friendly development.

Final Project, Unit 4

Project code:
https://github.com/prather-mcs/nano/tree/master/project-4

The final project for this unit required us to make important changes to a multi-page site to bring its PageSpeed Insights score (PSI was described above under "...What I Learned") from 30 to almost 100, while still perfectly preserving the original appearance and functionality of the site.

  • See the original, un-optimized version:
    https://prather-mcs.github.io/udacity-units/unit-4/initial-state
  • See the same page after it has been fully optimized:
    https://prather-mcs.github.io/udacity-units/unit-4/final-state

To accomplish this, I had make optimizations to (a) delivery of webfonts, (b) delivery of images, (c) delivery and sync/async behavior of JavaScript files, and (d) delivery of CSS style rules.

Furthermore, with this project, I took the opportunity to get started learning the Node runtime, for the purpose of minifying JS files. So the final state of the code on the live page contains minified JavaScript code, which was a PSI score improvement. For more practice with Node, I also minified the HTML and styling rules, although that was not measured by PSI. There is a lot more to say about Node, but for now I'll summarize it by saying that I consider it a very important toolkit which I value very much.

Finally, I also re-wrote the JavaScript animation routines on the "pizzeria" page, to change it from being extremely janky to perfectly fluid at 60 fps animation rate. Animation there occurred while you scroll the page and stops when you stop scrolling. The weaker your computer's graphics and CPU resources, the greater the difference you will notice due to my changes. (The linked images can be summarized by saying that I greatly reduced the amount of time spent executing scripts and re-painting the webpage, down from over 50 milliseconds per animation event to single-digit millisecond animations.)

Final Project, Unit 3

Project code:
https://github.com/prather-mcs/nano/tree/master/project-3

I wrote the object-oriented JavaScript to make this re-make of the classic arcade game "Frogger" work:

  • Play it now:
    https://prather-mcs.github.io/udacity-units/unit-3/index.html

Of course at this stage of the Nanodegree program, we were not required to write the whole game from scratch, nor to create our own image assets. All images were provided for us, and the code that powers the game was partially done as well.

The base code would only draw a blank screen, so we had to make decisions like "how big should the road be" and write code to draw the road on the canvas. Then we had to figure out how to draw the moving obstacles (cars, bugs) at an appropriate speed moving from left to right. Then we had to figure out how to have a player sprite which accepted player inputs and would move accordingly. Collision detection was also completely up to us to implement. And in creating our solutions for these design specifications, we had to use both JavaScript's prototypal inheritance and its function closure rules correctly, to come up with an efficent "object-oriented" implementation for the game engine.

My final game, as linked above, implements only two gameplay features beyond the basic requirements. I allowed pausing the game with a button press, and I implemented collision detection on the cars themselves, such that they would bump into each other, and not be able to pass through slower cars in front of them. I definitely had some great ideas of improvements to make, to make the game actually fun to play, but I noticed several other students do it first:

  • http://alexandrk.github.io/frontend-nanodegree-arcade-game
  • http://arcadegame2000.appspot.com
  • http://rachkoud.github.io/frontend-nanodegree-arcade-game/index.html
  • http://anjalinanda.github.io/FroggerGame

So I decided to not spend the significant amount of extra time it would take to make a truly fun game, and make it playable on mobile devices, which it really needs to be. My playable version might not be the most impressive thing. However, the feature I implemented where cars have collision detection on themselves and cannot pass through each other is something not one of the other versions did.

What ought to be impressive is my code behind the game. Not only did I take the time to make it truly efficient through prototypally-inherited common functions wherever possible, but I also put in a lot of time to follow JSDoc commenting rules properly. The JSDoc comment format is as syntactical as a programming language itself, because programs parse JSDoc comments to provide static analysis and automatic documentation. So my code comments here are another programming achievement.

On the other hand, I now know more about how code is really organized in real-world projects, and there are some shortcomings in this code. I'm talking about the use of global variables, shared by three three separate .js script files. That was basically given to us in that way, with nothing said about changing it or it needing to be changed. I would always want to build code in the modular design pattern (used with require or import), and not rely on global variables external to a script. Or internal to a script, for that matter. I know how to remove global variables altogether. So when I say I'm proud of my code, I just mean that I made it neat and efficient within the framework given to me, even if it's not a realistic framework. Also, using JSDoc should be recognized as a plus.

Unit 2: Basic JavaScript

Overview of Courses and What I Learned

  • Udacity Course: JavaScript Basics

    This course introduced the basic JavaScript syntax, data structures, and functions.

    I had decent experience with these in Python, but I was actually totally new at JavaScript before this course.

    After completing future projects and immersing myself daily in web develoment content, I am now fully fluent in JavaScript, very confident with it, and it is my go-to general purpose programming language, thanks to Node. I have a Linter in my text editor to save time by catching syntax typos immediately. And I automate my workflow wherever possible, sometimes with powerful packages like Grunt and Yeoman, and sometimes with custom scripts I wrote myself.

    And all of those that is well beyond the scope of this original course. I sure have come a long way since this class.

  • Udacity Course: Intro to jQuery

    jQuery is a great library to perform DOM-manipulating tasks more safely (in the sense of cross-browser functionality) and quickly than coding up from scratch in Vanilla JavaScript.

    This course was a decent introduction to jQuery, showing you something like 30 different functions from the library, and how to use them.

    At this stage in my development as a proficient web developer, I study the jQuery source code in my spare time, to see how it functions at the Vanilla JS level.

    Finally, one of the first "programs" I ever built I called "jQuery Sandbox". It's not a full program, of course, but what it does is provide an immediate read-eval-print console environment with the jQuery library, so you can test code that requires jQuery immediately, in an environment that contains all of jQuery, but nothing besides.

    You can check out jQuery Sandbox here:
    https://prather-mcs.github.io/jQuerySandbox/index.html

    You'll be able to perform commands like
    $('body');
    or
    $('body').append('<img src="http://maps.googleapis.com/maps/api/streetview?size=600x400&location=Universal+Studios">');
    immediately from your browser's console.
